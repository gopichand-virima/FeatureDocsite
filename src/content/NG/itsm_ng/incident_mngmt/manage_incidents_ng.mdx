# Manage Incidents

This comprehensive guide covers all aspects of managing incidents from assignment through resolution and closure.

## Incident List Overview

The main incident list provides a comprehensive view of all incidents in the system.

### List View Features

#### Column Customization
Customize which columns display:
1. Click the **Columns** button
2. Check columns to show
3. Uncheck columns to hide
4. Drag to reorder
5. Click **Save View**

#### Standard Columns
- **Number**: Unique incident identifier
- **Short Description**: Brief summary
- **Status**: Current state
- **Priority**: P1 (Critical) through P4 (Low)
- **Assigned To**: Current owner
- **Created**: Date/time opened
- **Updated**: Last modification
- **SLA**: Time remaining or breach status

### Filtering Incidents

#### Quick Filters
Predefined filter options:
- **My Open Incidents**: Assigned to you, not closed
- **My Group's Incidents**: Assigned to your group
- **High Priority**: P1 and P2 incidents
- **SLA Violations**: Breached or at risk
- **Unassigned**: No owner assigned
- **Recent**: Created in last 24 hours

#### Custom Filters
Create your own filters:

1. Click **Filter** button
2. Add filter conditions:
   - Status equals "In Progress"
   - Priority equals "High"
   - Assigned Group equals "Network Team"
3. Click **Apply**
4. Optionally save filter for reuse

#### Advanced Filtering
Build complex filters:
- Use AND/OR logic
- Multiple conditions
- Date ranges
- Text contains/equals
- Number comparisons

### Sorting

Sort incidents by any column:
1. Click column header to sort ascending
2. Click again to sort descending
3. Hold Shift and click multiple columns for multi-level sorting

### Grouping

Group incidents by common attributes:
1. Click **Group By** dropdown
2. Select grouping field:
   - Status
   - Priority
   - Assigned Group
   - Category
3. Incidents organize into collapsible groups

## Working with Incidents

### Claiming Incidents

Pick up unassigned incidents:

1. Navigate to **Unassigned Queue**
2. Find appropriate incident
3. Click the incident to open
4. Click **Assign to Me**
5. Begin work on incident

### Updating Incidents

Make changes to incident details:

1. Open the incident
2. Click **Edit** button
3. Modify fields as needed
4. Add work notes about changes
5. Click **Save**

**Commonly Updated Fields:**
- Status
- Assigned To
- Priority
- Category
- Resolution notes

### Adding Work Notes

Document work performed:

1. Open incident
2. Navigate to **Work Notes** tab
3. Click **Add Work Note**
4. Enter details of work performed
5. Notes visible to support team only
6. Click **Save**

**Best Practices:**
- Document all troubleshooting steps
- Record time spent
- Note findings and results
- Include next steps
- Keep notes professional

### Communicating with Users

Add comments visible to end users:

1. Open incident
2. Navigate to **Comments** tab
3. Click **Add Comment**
4. Write user-friendly message
5. Comment sent via email to requester
6. Click **Save**

**Comment Tips:**
- Use plain language (avoid jargon)
- Explain what you're doing
- Set expectations for resolution
- Request additional information if needed
- Remain professional and courteous

## Incident Status Management

### Status Lifecycle

Incidents progress through these states:

#### New
- Just created
- Awaiting assignment
- No work begun
- SLA clock started

#### Assigned
- Assigned to person or group
- Awaiting work to begin
- Ownership established

#### In Progress
- Actively being worked
- Investigation underway
- Troubleshooting in progress
- Updates being provided

#### On Hold
- Temporarily paused
- Waiting for information
- Awaiting third party
- SLA clock may pause (configurable)

**When to Use On Hold:**
- Waiting for user response
- Vendor support required
- Change must be completed first
- External dependencies

#### Resolved
- Solution implemented
- Waiting for user confirmation
- Monitoring period
- SLA resolution timer stopped

#### Closed
- User confirmed resolution
- Incident fully resolved
- No further action needed
- Historical record only

#### Cancelled
- Incident withdrawn
- Duplicate of another incident
- Created in error
- No longer relevant

### Changing Status

Update incident status:

1. Open the incident
2. Click **Status** dropdown
3. Select new status
4. Add work note explaining status change
5. Click **Save**

**Status Change Rules:**
- Cannot skip required statuses
- Some changes require approval
- Automated notifications sent
- Audit trail maintained

## Incident Resolution

### Resolution Process

1. **Identify Solution**
   - Troubleshoot the issue
   - Research knowledge base
   - Consult documentation
   - Engage specialists if needed

2. **Implement Solution**
   - Apply fix or workaround
   - Test functionality
   - Verify resolution
   - Document steps taken

3. **Update Incident**
   - Change status to "Resolved"
   - Enter resolution notes
   - Document solution details
   - Update resolution code

4. **User Verification**
   - User confirms fix works
   - User closes incident
   - Or auto-close after time period

### Resolution Fields

When resolving, complete:

#### Resolution Notes *
Detailed description of solution:
- What was done
- Why it fixed the issue
- Any follow-up needed
- Preventive measures

#### Resolution Code *
Category of resolution:
- **Fixed**: Issue permanently resolved
- **Workaround**: Temporary solution provided
- **Cannot Reproduce**: Issue not found
- **No Issue Found**: Working as designed
- **Duplicate**: Same as another incident
- **Cancelled**: Requester withdrew

#### Resolution Time
Automatically captured:
- Total time to resolve
- Used for SLA reporting
- Metrics and trending

### Knowledge Article Creation

Create knowledge from resolution:

1. In resolved incident
2. Click **Create Knowledge Article**
3. System pre-fills from incident
4. Edit and enhance content
5. Add troubleshooting steps
6. Include screenshots
7. Publish to knowledge base

**When to Create Knowledge:**
- Common or recurring issues
- Complex resolutions
- Novel problems
- Requests for documentation

## Escalation

### When to Escalate

Escalate incidents when:
- Resolution exceeds your skill level
- SLA at risk of breach
- User requests escalation
- Manager involvement needed
- Third-party support required

### Escalation Types

#### Functional Escalation
Transfer to higher skill level:

1. Open incident
2. Click **Escalate**
3. Select escalation group
4. Add escalation notes
5. Explain issue and steps taken
6. Click **Save**

**Escalation Path Example:**
1. Level 1 Support (0-15 min)
2. Level 2 Support (15-60 min)
3. Level 3/Specialists (60+ min)
4. Vendor Support (if required)

#### Hierarchical Escalation
Escalate to management:

1. Open incident
2. Click **Escalate to Manager**
3. Select manager
4. Explain need for escalation:
   - SLA breach risk
   - User dissatisfaction
   - Resource constraints
   - Complex issue
5. Manager notified immediately

### Escalation Best Practices

- Provide complete handoff notes
- Include troubleshooting already done
- Attach relevant logs or screenshots
- Set user expectations
- Follow up on escalated incidents

## Bulk Operations

### Bulk Updates

Update multiple incidents at once:

1. Select incidents using checkboxes
2. Click **Bulk Update**
3. Choose update type:
   - Assign to user/group
   - Change status
   - Update priority
   - Add work notes
   - Change category
4. Enter new value
5. Confirm changes
6. System updates all selected incidents

### Bulk Assignment

Assign multiple incidents:

1. Filter for unassigned incidents
2. Select incidents to assign
3. Click **Assign**
4. Choose assignment method:
   - Assign to specific person
   - Assign to group
   - Distribute evenly among team
5. Confirm assignment

## Incident Relationships

### Linking Related Incidents

Connect related incidents:

1. Open incident
2. Navigate to **Related Records** tab
3. Click **Add Relationship**
4. Search for related incident
5. Select relationship type:
   - **Parent/Child**: One incident causes another
   - **Related**: Connected but independent
   - **Duplicate**: Same issue
6. Save relationship

### Parent-Child Incidents

For incidents caused by same root issue:

**Parent Incident:**
- Main incident tracking root issue
- May be linked to problem record
- Resolving parent may resolve children

**Child Incidents:**
- Individual user reports of same issue
- Link to parent incident
- May close automatically when parent resolved

### Duplicate Incidents

Mark duplicates:

1. Identify duplicate incident
2. Open the duplicate
3. Click **Mark as Duplicate**
4. Search for original incident
5. Select original
6. Add notes
7. Duplicate incident closed
8. Requester notified of original incident number

## SLA Management

### Viewing SLA Status

Monitor SLA compliance:

**List View:**
- SLA column shows time remaining
- Color coded:
  - Green: On track
  - Yellow: At risk (< 25% remaining)
  - Red: Breached

**Detail View:**
- SLA section shows:
  - Target resolution time
  - Time elapsed
  - Time remaining
  - Breach status

### SLA Breaches

When SLA breached:

1. **Immediate Actions:**
   - Priority escalation (automatic)
   - Management notification
   - Incident flagged
   - Enhanced monitoring

2. **Required Follow-Up:**
   - Document reason for breach
   - Expedite resolution
   - Keep user informed
   - Post-incident review

### Pausing SLA

SLA may pause automatically when:
- Status changed to "On Hold"
- Waiting for user response
- External vendor involved
- After business hours (configurable)

## Reporting

### Standard Reports

Access pre-built reports:

1. Navigate to **Reports** > **Incident Management**
2. Select report:
   - **Incident Volume**: Total incidents by period
   - **Resolution Time**: Average time to resolve
   - **SLA Compliance**: % meeting SLA
   - **Category Analysis**: Incidents by type
   - **Team Performance**: Agent statistics

### Custom Reports

Create custom reports:

1. Click **New Report**
2. Select data source: Incidents
3. Choose fields to include
4. Add filters
5. Select grouping
6. Choose chart type
7. Save and schedule
