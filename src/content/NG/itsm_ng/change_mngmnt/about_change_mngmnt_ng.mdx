# Overview of Change Management

The Project Change Management Process is followed during the Execution phase of the Project Management Life Cycle once the project has been formally defined and planned.

## What is the Change Management Process

The Change Management Process is a method by which changes to the project (for example, to the scope, deliverables, timescales, or resources) are formally defined, evaluated, and approved before implementation. The process entails completing various control procedures to ensure that, if implemented, the change will cause minimal impact on the project\'s objectives.

The Change Management Process is used to ensure that every change identified is:

::: Bullet1
Communicated

::: Bullet1
Documented

::: Bullet1
Reviewed

::: Bullet1
Approved

::: Bullet1
Implemented

## When to use the Change Management Process?

Any change to the project during the Execution phase will need to be formally managed as part of the Change Management Process. Without a formal Change Management Process, the objective of delivering a solution - within time, cost and quality - may be compromised.

The Change Management Process is terminated when the Execution phase of the project is completed (such as just before project closure).

## Change Management Process

### Overview

The Change Management Process ensures that each change introduced to the project environment is appropriately defined, evaluated, and approved before implementation.

Change Management will be introduced to this project through the implementation of five key processes:

::: Bullet1
The submission and receipt of change requests.

::: Bullet1
The review and logging of change requests.

::: Bullet1
The determination of the feasibility of change requests.

::: Bullet1
The approval of change requests.

::: Bullet1
The implementation and closure of change requests.

### Identify and Submit Change Request

This process allows any project team member to submit a request for a change to the project.

The Change Requester:

::: Bullet1
Identifies a requirement for a change to any aspect of the project (for example, scope, deliverables, timescales and organization).

::: Bullet1
Completes a Request For Change (RFC) and distributes the form to the Project Manager.

The RFC summarizes the change:

::: Bullet1
Description

::: Bullet1
Reasons

::: Bullet1
Benefit.

::: Bullet1
Costs

::: Bullet1
Impacts

::: Bullet1
Any supporting documentation

::: Bullet1
Approvals

### Review Change Request

The Project Manager reviews the RFC and determines whether or not additional information is required for the Change Advisory Board to assess the full impact of the change on the project time, scope, and cost.

The decision will be based on factors, such as:

::: Bullet1
Number of change options presented.

::: Bullet1
Feasibility and benefits of the change.

::: Bullet1
Complexity and/or difficulty of the change options requested.

::: Bullet1
Scale of the change solutions proposed.

The Project Manager will record the RFC details in the Change Log to track the status of the change request.

## Approve Change Request

The Project Manager will forward the Change Request Form and any supporting documentation to the Change Advisory Board or Emergency Change Advisory Board (ECAB) for review and final approval.

They will determine the feasibility of this change by examining factors, such as:

::: Bullet1
Risk to the project in implementing the change.

::: Bullet1
Risk to the project in NOT implementing the change.

::: Bullet1
Impact on the project in implementing the change (time, resources, finance, quality).

After a formal review, they may:

::: Bullet1
Reject the change.

::: Bullet1
Request more information related to the change.

::: Bullet1
Approve the change as requested.

::: Bullet1
Approve the change subject to specified conditions.

### Implement and Close Change Request

If the change is approved, the following will occur:

::: Bullet1
An implementation date of the change will be identified.

::: Bullet1
A test of the change will be scheduled and performed.

::: Bullet1
The change will be implemented.

::: Bullet1
The implementation of the change will be reviewed and deemed successful or corrective actions will be taken.

::: Bullet1
The success of the change implementation will be communicated to all parties.

::: Bullet1
The change request will be closed on the Change Log.

## Change Management Roles

The following will play a role in the request, review, tracking, and approval of a change request:

### Change Requester

The Change Requester initially recognizes a need for change to the project and formally communicates this requirement to the Project Manager. The Change Requester is responsible for the following:

::: Bullet1
Identifying the requirement to make a change to the project.

::: Bullet1
Documenting the requirement by completing an RFC.

::: Bullet1
Submitting the RFC to the Project Manager for review.

### Project Manager

The Project Manager receives, logs, monitors, and controls the progress of all changes within a project. The Project Manager is responsible for the following:

::: Bullet1
Receiving all RFCs and logging them in the Change Log.

::: Bullet1
Categorizing and prioritizing all RFCs.

::: Bullet1
Reviewing all RFCs to determine whether additional information is required.

::: Bullet1
Forwarding the RFC to the Change Control Board for approval.

::: Bullet1
Escalating all RFC issues and risks to the Change Control Board.

::: Bullet1
Reporting and communicating all decisions made by the Change Control Board.

::: Bullet1
Close the Change Request and update the Change Log when completed.

### Change Control Board

The Change Advisory Board (CAB) is the Steering Committee or other authorized body who is the principle authority for all RFCs forwarded by the Project Manager. The Change Advisory Board Group is responsible for the following:

::: Bullet1
Reviewing all RFCs forwarded by the Project Manager.

::: Bullet1
Considering all supporting documentation.

::: Bullet1
Approving/rejecting each RFC based on its relevant merits.

::: Bullet1
Resolving change conflict (where two or more changes overlap).

::: Bullet1
Identifying the implementation timetable (for approved changes).

### Project Team/Project Manager

The Project Team and Project Manager schedule and implement all changes. This group is responsible for the following:

::: Bullet1
Scheduling all changes (within the parameters provided by the Change Advisory Board).

::: Bullet1
Testing all changes before implementation.

::: Bullet1
Implementing all changes within the project.

::: Bullet1
Reviewing the success of a change, following implementation.

## Change Management Documents

The following documentation is used to monitor and/or control changes to the project.

::: Bullet1
The **Change Request Form** is used to identify and describe a proposed change to the project.

::: Bullet1
The **Change Log** is the log where all requests for changes are registered and tracked through to resolution.
